# Kaizen ÔÇö Architecture Review (2026-06)

A deep read of the codebase end to end: what is solid, what is thin, what will
bite, and a prioritized path to the stated goal ÔÇö *an always-on personal agent
that is actually always on.*

Scope: `kaizen/` (~3.2k LOC across 53 modules), `tests/` (now 17 files / 108
tests), `deploy/`, and `docs/`. Reviewed at branch `claude/deepen`.

---

## 1. Executive summary

Kaizen is a genuinely well-structured agent skeleton. The seams are in the right
places: a headless core, a `Provider` protocol that makes cloud/local/subscription
backends interchangeable, a `MemoryStore` protocol with two implementations, a
single approval gate fronting every self-state mutation, and a four-layer persona
stack. The documentation (14 ADRs + architecture + design-log) is unusually
thorough for a project this size and the design intent is legible from the code.

The gap between Kaizen and its tagline is not architecture ÔÇö it is **wiring and
operational completeness**. Three things stand between "runnable skeleton" and
"always on":

1. **The production entrypoint does not run the always-on surface.** The
   container starts the interactive CLI REPL, which cannot survive a detached
   container. (Section 4.1 ÔÇö the single highest-priority fix.)
2. **The pgvector memory path is implemented but never exercised** ÔÇö no
   integration test, no migration story, embedding happens inline on the write
   path. (Section 4.2.)
3. **The "thinks before it speaks" intelligence is heuristic stubs** ÔÇö triage,
   tone, relevance governor, and the curator's extraction are all keyword
   matchers with the model-backed versions still TODO. The seams exist; the
   intelligence does not yet. (Section 4.4.)

This review fixes the most measurable of these (triage, via an eval harness) and
lays out the rest as a roadmap.

## 2. Strengths

### 2.1 Clean provider abstraction
`kaizen/providers/base.py` defines a tiny `Provider` protocol (`name`, `tier`,
`async complete`). Four backends implement it without the loop knowing which is
in play: `mock.py` (deterministic, zero-infra), `local.py` (Ollama),
`anthropic.py` (metered API), `claude_code.py` (subscription via CLI subprocess).
The message-conversion functions (`to_anthropic`, `to_ollama`, `flatten_prompt`)
are pure and SDK-free, so they are unit-tested without keys or network ÔÇö this is
exactly right and is the reason the test suite runs offline.

### 2.2 The approval gate is a real chokepoint, not theater
`kaizen/safety/gate.py` is the single path to any self-state mutation, and
`kaizen/curator/proposals.py` *composes* it rather than duplicating its
bookkeeping (`ProposalQueue.__init__` takes a `gate=`). The dedup logic
(`_fingerprint` + `_by_fingerprint`) correctly folds repeated curation runs onto
one pending card and lifts confidence to the max ÔÇö a thoughtful detail that
prevents the operator from drowning in duplicate proposals when the curator runs
every turn.

### 2.3 Memory and instinct models share a shape
`Fact` (`memory/base.py`) and `Instinct` (`curator/instinct.py`) both carry
`confidence` / `source` / `first_seen` / `last_seen`. That consistency means the
decay/conflict-resolution logic the ADRs promise can be written once against a
common shape.

### 2.4 Lessons are encoded, not just documented
`kaizen/tools/youtube.py` is the standout: it encodes three concrete failures
from a prior project (`ytmerge`) directly into the type system and control flow ÔÇö
`RunLocation.RESIDENTIAL` so the orchestrator never runs it from a datacenter IP,
exception classification by *type name* (robust across library versions), and a
`TransientToolError` that distinguishes "blocked, retry" from "no captions." This
is institutional memory turned into code.

### 2.5 Deploy scaffolding is thorough
`deploy/` is more complete than most hobby projects ever get: a hardening
`provision.sh`, systemd units, a restic backup timer, a documented restore drill,
loopback-only data-plane binding with Tailscale for admin, and a runbook with a
failure-mode table. The cost model is worked out to the dollar.

### 2.6 Persona is layered and testable
The four-layer voice stack (`persona/prior.py` floor, curator-approved learned
traits, `persona/tone.py` per-turn register, router voice-tier pin) is cleanly
separated, and the tone classifier resolves conflicts by an explicit priority
order with a documented rationale (anger beats sarcasm because mis-reading anger
is the worse failure).

## 3. Weaknesses

### 3.1 Test isolation depended on a clean environment (fixed this session)
The integration tests called `build_agent()` with no explicit `Settings`, so they
inherited the operator's ambient `.env` / `KAIZEN_*` vars. On a configured box
(`KAIZEN_USE_LOCAL_MODEL=true`), the mock LOCAL provider was silently swapped for
a real Ollama client and the background scribe tried to hit a non-existent
endpoint ÔÇö three integration tests failed locally while passing in clean CI.
Fixed by `tests/conftest.py`: an autouse fixture strips `KAIZEN_*` and disables
`.env` discovery, so every test runs against documented zero-config defaults.

### 3.2 Heuristic intelligence with no measurement (triage fixed this session)
`triage()`, `classify_tone()`, `InterjectionGovernor._relevance()`, and
`Curator._extract_from_message()` are all keyword/regex heuristics. That is a
reasonable starting point ÔÇö but until this session none had an eval, so there was
no way to know whether a change improved or regressed routing. The old `triage()`
in particular had a latent defect (Section 4.4) that no test caught.

### 3.3 Inline embedding on the memory write path
`PostgresMemoryStore.add_fact` (`memory/postgres.py:62`) awaits
`self._embedder.embed(...)` ÔÇö an HTTP round-trip to Ollama ÔÇö inside the write. The
scribe runs in a background task so this does not block replies today, but it
couples fact-writes to embedder availability and latency, and there is no
batching. At backfill scale (ADR 0012 talks about millions of messages) this is
the wrong shape.

### 3.4 `search` ignores recency, confidence, and decay
Both stores rank purely by lexical overlap (in-memory) or cosine distance
(Postgres). The `Fact` model carries `confidence` / `last_seen` and the ADRs
promise conflict-resolution-by-recency and decay, but no store implements it. A
stale, low-confidence fact ranks identically to a fresh, high-confidence one.

### 3.5 In-memory-only state for gate, queue, identity, and skills
`ApprovalGate`, `ProposalQueue`, `IdentityGraph`, and `SkillRegistry` are all
process-local dicts. For an "always-on" agent, an approved instinct or a learned
skill evaporating on restart is a correctness problem, not just an inconvenience.
The interfaces are clean enough to back with Postgres later, but nothing does yet.

### 3.6 No internal API; surfaces import the core directly
ADR 0004 / the architecture doc describe a headless core exposing an *internal
API* that surfaces are thin clients of. In reality `discord_bot.py` imports
`build_agent` from `cli.main` and shares the process. The "move a conversation
from Discord to the terminal mid-thought" promise (README) is not achievable
while each surface builds its own in-process `AgentBundle` with its own sessions.

### 3.7 Sessions are unbounded and never persisted
`Session.messages` is an unbounded list; `context.build` slices `recent(20)` for
the prompt but the session itself grows without limit, and nothing persists or
restores it. Compression is named in the docstring as "arrives later."

## 4. Risks and concrete findings (with file paths)

### 4.1 CRITICAL ÔÇö production container runs the interactive CLI, not the bot
`Dockerfile:54` sets `ENTRYPOINT ["python", "-m", "kaizen"]`, which runs
`kaizen/cli/main.py:run` Ôćĺ `_chat()` Ôćĺ a `while True` loop blocking on
`input("you> ")`. `deploy/docker-compose.box-a.yml` (the `kaizen` service, line
99) provides **no `command:` override**, so the always-on production service
launches the *interactive REPL*. In a detached container there is no stdin;
`input()` raises `EOFError`, `_chat` breaks, the process exits, and
`restart: unless-stopped` turns it into a crash loop that never serves Discord.

The always-on surface is `python -m kaizen.surfaces.discord_bot`. **Fix:** either
add `command: ["python", "-m", "kaizen.surfaces.discord_bot"]` to the compose
service, or introduce a `KAIZEN_SURFACE` env switch in `__main__.py` that selects
CLI vs Discord. This is the single thing most directly blocking "actually always
on."

### 4.2 HIGH ÔÇö pgvector path was untested (integration test added; migration still pending)
`memory/postgres.py` is well-written but had zero test coverage, and `init_db()`
uses `Base.metadata.create_all` (the module docstring acknowledges Alembic "comes
later"). This is now partly closed: a live `tests/integration` suite (gated on
`KAIZEN_TEST_DATABASE_URL`, skipped otherwise) proves a fact round-trips through
embed Ôćĺ store Ôćĺ cosine search and that the ambient-scribe write path persists and
recalls against a real `pgvector/pgvector:pg16` container. Embeddings use a new
deterministic `HashEmbedder` so the test needs no model, GPU, or API key. A
one-command `scripts/pgvector-test.sh` does up Ôćĺ migrate (`CREATE EXTENSION`) Ôćĺ
test Ôćĺ down. **Still open:** `EMBED_DIM` is hard-coded to 768 and the schema is
still `create_all`; the Alembic baseline migration that pins the dimension as a
versioned change (below) has not yet landed.

### 4.3 HIGH ÔÇö curator extraction is regex-only; LLM seam is a comment
`curator/review.py:132` `_extract_from_message` is three regexes. The docstring
says "the real value comes from an LLM-backed extractor plugging into the same
shape (provider call goes where `_extract_from_message` is now)" ÔÇö but the
`Curator` dataclass holds no provider and the method is sync. The seam is
*described* but not *built*: adding a provider would change the method to async
and ripple into `review()`. Worth doing deliberately rather than as an
afterthought.

### 4.4 MEDIUM ÔÇö triage MODERATE class was unreachable (fixed this session)
The old `triage()` escalated to HARD on any substring match of a hard hint
(`"why" in text` fired on `"why not"`) and to MODERATE only on length > 120
chars. The new **eval harness** (`kaizen/eval/`) graded it against 26 golden
fixtures and surfaced the defect quantitatively: **MODERATE precision and recall
were both 0%** ÔÇö the heuristic literally could not route a realistic
moderate-difficulty message to the cheap tier; everything went LOCAL or FRONTIER.
Overall accuracy was 69%.

Rewriting `triage()` into a layered signal scorer (whole-word hard hints, a
length floor for soft hints like *why*/*explain*, and positive MODERATE signals
for synthesis verbs) raised it to **96% accuracy with MODERATE at 100%
precision / 83% recall**, measured against the same fixtures. The one remaining
miss ("draft ÔÇŽ and why" Ôćĺ HARD) over-routes in the *safe* direction and is left
documented rather than over-fit. See Section 6.

### 4.5 MEDIUM ÔÇö redaction runs nowhere on the live path
`safety/redact.py` is solid and tested, but `grep` shows `scrub_message` /
`redact` are not called by `scribe.py`, `context.py`, or any surface. The module
docstring claims "the scribe and identity graph run `redact` on incoming
content" ÔÇö they do not. Credentials pasted into a conversation would reach the
memory store unredacted. **Fix:** call `redact` in `Scribe._extract` before the
transcript is sent to the model and before facts are stored.

### 4.6 LOW ÔÇö `claude_code.py` shells out per request with no concurrency guard
`ClaudeCodeProvider.complete` spawns a `claude` subprocess per call. The
subscription has rate limits (acknowledged in the docstring) and there is no
semaphore, queue, or backpressure. Fine for a single interactive operator;
a hazard if the Discord surface fans out.

### 4.7 LOW ÔÇö `_relevance` / in-memory `search` tokenization is naive
Both split on whitespace with no stemming, casefolding edge cases, or stopword
handling (the curator has a stopword set; the governor and in-memory store do
not). Acceptable for a stub; will produce surprising routing/recall as soon as
real traffic hits it.

## 5. What this session changed

1. **Test isolation** ÔÇö `tests/conftest.py` makes the suite hermetic; the 3
   environment-dependent integration failures are fixed at the root cause.
2. **Typing is now a hard gate** ÔÇö `[tool.mypy]` in `pyproject.toml` runs a
   strict config (`disallow_untyped_defs`, `check_untyped_defs`,
   `warn_return_any`, etc.) over `kaizen/` with no per-module blanket ignores;
   CI (`.github/workflows/ci.yml`) runs `mypy` as a required step (the `|| true`
   soft-fail is gone). 19 real annotation gaps were fixed (notably: `MemoryStore`
   protocol was missing the `.name` attribute both implementations have and the
   CLI relies on; `ToolRegistry.list` / `SkillRegistry.list` shadowed the `list`
   builtin in their own return annotations).
3. **Architecture diagram** ÔÇö a Mermaid flowchart in `README.md` (validated by
   `mermaid-cli`) showing surfaces Ôćĺ core Ôćĺ tiers, the async learning path, and
   the approval gate.
4. **Triage eval harness** (the substantive improvement) ÔÇö `kaizen/eval/`:
   golden fixtures, a scoring harness with confusion matrix and per-class
   precision/recall, a CLI runner (`python -m kaizen.eval`) that doubles as a
   regression check, and `tests/test_eval.py` (13 tests). It drove a measured
   `triage()` improvement from 69% Ôćĺ 96% accuracy.
5. **pgvector path proven end to end** (P1) ÔÇö a deterministic, dependency-free
   `HashEmbedder` (`kaizen/memory/embedder.py`) selected via `KAIZEN_EMBEDDER`,
   a live `tests/integration` suite (3 tests) that writes facts, similarity-
   searches, and exercises the ambient-scribe write path against a real
   `pgvector/pgvector:pg16` container, and a one-command reproducible cycle
   (`scripts/pgvector-test.{sh,ps1}` + `deploy/docker-compose.pgvector-test.yml`).
   The suite skips cleanly when no DB is configured, so the offline run stays at
   108 green.

## 6. The triage eval harness (improvement detail)

**Why triage.** Routing is the cost/quality lever of the whole system. Every
message is scored by `triage()` and sent to LOCAL (free), CHEAP, or FRONTIER
(expensive). Under-routing sends a hard question to a weak model; over-routing
burns frontier dollars on a greeting. It is the highest-leverage heuristic in the
codebase and was the only one with a defect that no test caught.

**Design.** `kaizen/eval/harness.py` defines `Triager = Callable[[Message],
Difficulty]` ÔÇö the *same* signature `triage` already has and that a future
local-classifier triager will have. `evaluate(triager, cases)` returns an
`EvalReport` with `accuracy`, a `fair_accuracy` that excludes cases tagged as
known heuristic blind spots, `confusion()`, `precision_recall()`, and a
human-readable `render()`. The golden set (`kaizen/eval/cases.py`) is deliberately
adversarial: it includes hard questions with no trigger keyword and easy asides
that happen to contain one, so the harness reports honest sub-100% numbers
instead of grading the heuristic against its own behavior.

**Result.** The harness exposed MODERATE at 0%/0% P/R. The rewritten layered
scorer in `kaizen/orchestration/router.py` reaches 96% accuracy (MODERATE
100%/83%). The CLI runner enforces an 80% floor so a future change to routing is
measured, not guessed ÔÇö which is the precondition for safely swapping the
heuristic for a model.

## 7. Prioritized roadmap to "actually always on"

Ordered by leverage. Each item names the files it touches.

### P0 ÔÇö Make the always-on surface actually run
- **Fix the production entrypoint** (Section 4.1). Add `command:` to the
  `kaizen` service in `deploy/docker-compose.box-a.yml` pointing at
  `kaizen.surfaces.discord_bot`, or add a `KAIZEN_SURFACE` switch in
  `kaizen/__main__.py`. Add a smoke test that imports the Discord surface and
  asserts `build_agent()` wires without a TTY. *Without this, nothing else
  matters ÔÇö the core never stays up.*

### P1 ÔÇö Persistent memory, proven end to end
- **Dockerized pgvector integration test.** *(Done this session.)* The
  `integration`-marked `tests/integration` suite runs `add_fact Ôćĺ search` (and the
  ambient-scribe write path) against `memory/postgres.py` when
  `KAIZEN_TEST_DATABASE_URL` is set, asserting the nearest fact comes back, and is
  skipped when no DB is present so the offline suite stays green. A deterministic
  `HashEmbedder` keeps it key-free; `scripts/pgvector-test.{sh,ps1}` and
  `deploy/docker-compose.pgvector-test.yml` make the up Ôćĺ migrate Ôćĺ test cycle
  one command.
- **Alembic baseline migration** replacing `create_all`, with `EMBED_DIM`
  pinned in the migration so an embed-model change is a versioned schema change,
  not silent corruption.
- **Move embedding off the write path** ÔÇö batch facts and embed asynchronously
  (`memory/postgres.py`, `memory/scribe.py`), so writes don't block on Ollama.
- **Implement recency/confidence ranking and decay** in `search` (Section 3.4),
  using the fields `Fact` already carries.

### P2 ÔÇö LLM-backed curator behind the mock seam
- Give `Curator` (`curator/review.py`) an optional `provider: Provider | None`.
  When present, `_extract_from_message` (made async) asks the model for
  structured preferences instead of running regexes; when absent, it falls back
  to the current regex path. Mirror `scribe.py`'s `parse_facts` defensive JSON
  parsing. Test against `MockProvider` so it stays offline-runnable. This is the
  "every cycle gets a little better at you" loop with real extraction.

### P3 ÔÇö Persist the always-on state
- Back `ApprovalGate`, `ProposalQueue`, `SkillRegistry`, and `IdentityGraph`
  with Postgres tables (Section 3.5). Approved instincts and learned skills must
  survive a restart, or the agent forgets everything it learned every deploy.

### P4 ÔÇö A real internal API and session continuity
- Introduce the headless-core service boundary the docs describe (Section 3.6):
  a small API the surfaces call instead of importing `build_agent`. Persist and
  key sessions so a conversation can move between Discord and the terminal ÔÇö the
  README's headline promise.

### P5 ÔÇö Extend the eval discipline to the rest of the heuristics
- Apply the `kaizen/eval/` pattern to `classify_tone` and the
  `InterjectionGovernor` (golden fixtures + accuracy floor), so the
  "thinks before it speaks" subsystems are measured before they are swapped for
  models ÔÇö the same way triage now is.

### P6 ÔÇö Wire redaction onto the live ingest path
- Call `safety/redact.py` in `Scribe._extract` and at the identity-graph
  ingress (Section 4.5), and add a test that a pasted `sk-ÔÇŽ` key never reaches
  the store.

## 8. Verdict

The skeleton is sound and the design is honest with itself ÔÇö the docstrings name
their own stubs and TODOs rather than overselling. The work to reach "always on"
is mostly *connecting* things that already exist (entrypoint Ôćĺ bot, store Ôćĺ
tested, heuristics Ôćĺ measured), plus the genuinely new work of an internal API
and LLM-backed extraction. The fixes in this session (hard typing gate, test
isolation, a measured routing improvement, and the eval discipline to keep it
honest) are the foundation that makes the P1ÔÇôP5 work safe to do.
